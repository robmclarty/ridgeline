Architecture Approaches

How to integrate design.md, concern detection, visual specialist, and tool families into the Ridgeline pipeline

A

Layered Artifact + Plugin Toolkits

Recommended. Three interconnected systems that compose cleanly.

Layer 1 — Concern Detection Engine
Extensible keyword→category registry scans shape.md. Categories: web-visual, game-visual, print-layout, etc. Each category maps to specialist activation + tool family. New categories added by dropping a config file.

Layer 2 — Design Artifact Lifecycle
ridgeline design command creates project-level .ridgeline/design.md via interactive shaping (reuses shaper pattern). Specifier's visual specialist reads project design + shape, produces feature-level design section. Builder/reviewer inherit both layers.

Layer 3 — Visual Tool Families (Plugin-based)
Each tool family is a Ridgeline plugin: plugin/web-visual/ bundles Playwright, axe-core, CSS analyzers, pixelmatch. Future: plugin/game-visual/ bundles asset validators, scene graph tools. Concern detection activates the right plugin. Reviewer invokes tools from the active plugin.

Pros

  • Each system is independently useful and testable
  • New tool families don't touch core pipeline code
  • Concern categories are data, not code
  • Plugin architecture already exists in Ridgeline

Cons

  • Three systems to build and coordinate
  • Plugin packaging adds complexity for tool dependencies (Playwright binary, etc.)
B

Monolithic Visual Pipeline

Single integrated system — concern detection, design artifact, and tools all built into core Ridgeline.

Everything lives in src/. Concern detection is a function in the pipeline. Design.md creation is a new command. Visual tools are bundled as npm dependencies of Ridgeline itself. The reviewer has hardcoded knowledge of which tools to run for which concerns.

Pros

  • Simpler to build initially
  • No plugin coordination overhead
  • Everything in one place

Cons

  • Ridgeline's install size balloons (Playwright alone is ~300MB)
  • Adding game/print tools means modifying core code
  • Violates Ridgeline's existing plugin-based extensibility model
  • Not extensible without code changes
C

Flavour-Driven (Everything in Flavours)

Visual concerns handled entirely by domain flavours — no new core infrastructure.

Each flavour (game-dev, web-frontend) defines its own visual specialist, reviewer prompts, and tool integrations. No shared concern detection — each flavour decides internally. Design.md is a flavour convention, not a core feature.

Pros

  • No core changes needed
  • Each domain fully controls its experience
  • Simplest initial implementation

Cons

  • Massive duplication — each flavour reimplements design.md handling
  • No shared design artifact lifecycle
  • Projects spanning domains (web game with print docs) can't compose
  • No way to detect concerns without choosing a flavour first
