End-to-End Pipeline Flow

How design.md, shape detection, visual specialist, and tool families integrate across the full Ridgeline pipeline

Pipeline — Non-Visual Project (no change from today)
ridgeline shape
→
scan shapes/ → no match
→
ridgeline spec (3 specialists)
→
plan → build → review
Unchanged. No visual specialist, no design.md, no visual tools.
Pipeline — Visual Project (new flow)
Phase 1 — Shape + Design Discovery
ridgeline shape
→
scan shapes/ → web-visual ✓
→
auto-chain: ridgeline design
→
shape.md + design.md
Design command checks for existing .ridgeline/design.md → reads as context if found
Asks design-focused questions informed by matched shape categories
Produces feature-level design.md (hard tokens + soft guidance, freeform markdown)
Phase 2 — Spec with Visual Specialist
ridgeline spec
→
4 specialists in parallel:
Completeness Clarity Pragmatism Visual Coherence ✦
→
Synthesizer
→
spec.md + constraints.md + taste.md
Visual specialist receives: shape.md + project design.md + feature design.md
Proposes visual acceptance criteria per feature, design-specific constraints
Synthesizer merges visual criteria into spec features alongside other perspectives
Phase 3 — Plan + Build (design.md carried forward)
plan
→
build phase N
Both receive design.md alongside constraints.md and taste.md
Builder treats hard tokens as mandatory, soft guidance as best-effort
Deviations noted in handoff.md (same pattern as taste.md)
Phase 4 — Review with Visual Verification
review phase N
→
Verification pipeline (graceful degradation):
● Code review against design.md always
● Screenshot capture → multimodal eval if playwright
● CSS audit (colors, sizes, complexity) if css-analyzer
● Accessibility check (WCAG violations) if axe-core
● Lighthouse audit (quality scores) if lighthouse
● Visual diff against references if pixelmatch + refs
→
verdict (with visual issues)
Hard token violations → blocking issues (review fails)
Soft guidance deviations → suggestions
Skipped tools → noted in verdict, never blocking
Alternate Entry — Standalone Design Command
ridgeline design
→
.ridgeline/design.md (project-level)
Run independently to establish or update project-level design system.
If design.md already exists, offers: refine / extend / use as-is.
All subsequent builds inherit this automatically.
